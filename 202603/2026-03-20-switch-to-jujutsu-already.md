# Switch to Jujutsu already

## TL;DR
Jujutsu (jj) 兼容 Git，采用可变提交模型。它摒弃暂存区，支持自动快照、非阻塞冲突及随时撤销，使代码历史灵活可塑，降低了心智负担，提升开发效率。

## Summary
**核心主题**
介绍版本控制系统 Jujutsu (jj) 的核心设计哲学与思维模型，对比 Git 的使用差异，阐述如何利用 jj 的可变提交机制简化开发流程并降低心智负担。

**关键信息**

*   **底层兼容性**
    Jujutsu 完全兼容 Git，可无缝对接 GitHub 等平台，允许在本地使用 jj 的强大功能，而对远程仓库表现为 Git 操作。
*   **核心思维模型差异**
    *   **Git (装配线模型)**：提交被视为不可变的实体。开发者在工作区修改代码，挑选部分内容暂存，最后打包成提交并封存，工作流呈现线性、不可逆的特征。
    *   **Jujutsu (橡皮泥模型)**：提交被视为可变的对象。开发者始终处于某个提交中，可以随时回到任意旧的提交进行修改、拆分或移动，代码历史像橡皮泥一样随时可塑。
*   **颠覆性机制设计**
    *   **提交可变**：任何未推送的提交都可以直接修改。编辑旧提交时，更改会自动实时持久化，无需重新创建 commit。
    *   **无暂存区**：不存在“未提交更改”的状态，用户始终“在提交中”，因此 Git 中的 stage 和 stash 概念在 jj 中不存在。
    *   **轻量级分支**：分支只是提交图中的节点关系。创建分支不需要命名（称为“书签”），命名仅用于与人沟通或 Push，不影响底层操作。
    *   **非阻塞冲突处理**：遇到冲突时系统不会停止工作流，仅标记冲突状态，允许开发者继续在其他地方工作，稍后统一修复，且修复会自动级联到后续提交。
    *   **自动快照与撤销**：系统会在操作或文件变更时自动创建快照。通过 Oplog (操作日志)，用户可以随时撤销任何操作或回退到仓库的任意历史状态，彻底避免数据丢失。

**逻辑脉络**

1.  **痛点与引入**
    作者因长期难以理解 Git 的复杂内部机制且操作恐惧，转向尝试 Jujutsu。通过 AI 辅助理解 jj 的思维模型后，不仅掌握了 jj，反而反向理解了 Git。
2.  **概念重塑**
    通过对比“装配线”与“橡皮泥”的比喻，详细解释了 jj 如何通过移除 Git 中的不一致性（如暂存区、不可变性），简化了底层模型。阐述了“始终在提交中”、“无 Stashing”及“提交自动保存”等核心特性。
3.  **工作流对比与实操**
    *   **分支管理**：Git 中分支是特殊的命名流；jj 中分支仅为图结构上的标签，只有在历史真正分叉（多个子节点）时才形成实际分支。
    *   **代码提交**：Git 需要暂存和新建提交；jj 允许随时通过 `describe` 添加消息，使用 `split` 拆分提交，或用 `absorb` 自动将修改吸收到相关历史提交中。
    *   **日常操作**：`edit` 代替 checkout（同时实现 soft reset），`abandon` 代替 hard reset，移动提交代替 cherry-pick，所有操作均可随时 `undo`。
4.  **工具集成与效率**
    推荐使用 `jjui` 可视化工具辅助理解提交图。提供自定义别名配置，展示了如何将“移动书签、创建新提交、推送”等繁琐步骤封装为简单命令。
5.  **价值总结**
    Jujutsu 虽未实现 Git 不能做的事，但极大地降低了操作摩擦。它鼓励开发者频繁尝试想法、修改历史和切换分支，而无需担心破坏代码库，从而从根本上改变了开发习惯和效率。
