# How I estimate work as a staff software engineer
- URL: https://www.seangoedecke.com/how-i-estimate-work/
- Added: 2026-03-18 01:39:32
- Tags: #engineering

## TL;DR
软件估值本质是政治工具，无法准确估算。工程师应关注未知工作风险，提供多种方案供管理层权衡，而非追求精确计算。

## Summary
核心主题：软件工作估值的本质与实用方法

关键信息与逻辑脉络：

1. 估值困境
   - 软件行业存在一种"礼貌的虚构"：认为可以准确估算项目时间
   - 实际上，**准确估算软件项目是不可能的**，因为项目主要由未知工作主导
   - 工程师常使用 T 恤尺码或"初始估值翻倍加 20%"等启发式方法来回避直接时间估算

2. 估值的真实目的
   - 估值并非由工程师做出，而是**政治工具**，供非工程人员用于项目决策
   - 管理层会根据业务需求对估值施加压力，高需求项目会被要求降低估值，低优先级项目则会被增加缓冲
   - 估值帮助管理者决定哪些项目获得资金，哪些被取消

3. 估值与工作的关系
   - 传统观念认为先有工作再估算时间，**实际情况相反**
   - 团队通常先有估值，再确定能在该时间内完成什么样的技术工作
   - 时间限制直接影响技术方案选择：充足时间会追求优雅设计，紧迫时间则专注于可行方案

4. 作者的估值方法
   - **先收集政治背景**：了解项目压力程度和管理层期望的估值范围
   - **带着估值查看代码**：思考"哪些方法能在一周内完成"，而非"完成需要多长时间"
   - **关注未知而非已知**：未知工作总是占项目大部分时间，估值需考虑代码库中的"黑暗森林"
   - **提供风险评估而非具体估值**：返回多种方案及其风险，让管理层权衡
   - **目标是匹配估值的可行方案**，而非精确计算工作量

5. 应对质疑
   - 拒绝估值是懦弱行为，会将估算责任推给非技术人员
   - 与管理层合作寻找技术妥协比持续对抗更有成效
   - 感受到估值压力的工程师通常处于"聚光灯下"，而未感受到的团队可能处于"边缘区域"

6. 总结观点
   - 软件估值普遍被误解，实际是管理者之间协商工作的工具
   - **准确估算软件工作是不可能的**，因为项目大部分时间都在处理未知问题
   - 好的估值应忽略已知工作，专注于评估未知问题的数量和风险程度
   - 工程师应建立信任，偶尔报告"不可能"的项目，以便在需要时被相信
