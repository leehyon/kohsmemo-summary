## TL;DR
Claude Code 采用六层架构，通过 Prompt Caching 实现代理循环，结合上下文分层、Skills 设计、工具策略、Hooks 机制和 Subagents 隔离，构建可验证、可审计的 AI 协作系统。

## Summary
核心主题：Claude Code 的架构、治理与工程实践

关键信息与逻辑脉络：

1. 系统架构六层模型
   - CLAUDE.md/rules/memory：长期上下文，定义"是什么"
   - Tools/MCP：动作能力，定义"能做什么"
   - Skills：按需加载的方法论，定义"怎么做"
   - Hooks：强制执行行为，不依赖模型判断
   - Subagents：隔离上下文的工作者，实现受控自治
   - Verifiers：验证闭环，确保输出可验、可回滚、可审计

2. 底层运行机制
   - 核心是代理循环：收集上下文→采取行动→验证结果→[完成或回到收集]
   - 五个关键层面：上下文表面、动作表面、控制表面、隔离表面、验证表面
   - Prompt Caching 是架构核心，影响成本结构和设计取舍

3. 上下文工程实践
   - 200K 上下文并非全部可用，固定开销占 15-20K
   - 推荐分层：常驻→CLAUDE.md，按路径加载→rules，按需加载→Skills，隔离加载→Subagents
   - 压缩机制陷阱：默认算法可能删除架构决策和约束理由
   - Plan Mode 价值：将探索与执行分离，降低错误假设修改概率

4. Skills 设计原则
   - 不是模板库，而是按需加载的工作流
   - 描述符要精简，每个 Skill 都占用上下文空间
   - 三种典型类型：检查清单型、工作流型、领域专家型
   - 反模式：描述过短、正文过长、一个 Skill 覆盖多件事、允许模型自动调用有副作用的 Skill

5. 工具设计策略
   - 好工具特征：名称明确、参数清晰、返回相关、边界清楚、成本可控
   - 设计原则：名称前缀分层、支持响应格式控制、错误信息包含修正建议
   - 演进教训：独立 AskUserQuestion 工具优于参数或格式约定

6. Hooks 应用场景
   - 适合：阻断修改、自动格式化、动态上下文注入、任务完成通知
   - 不适合：复杂语义判断、长时间业务流程、多步推理决策
   - 与 CLAUDE.md 和 Skills 形成三层治理体系

7. Subagents 使用方法
   - 核心价值是隔离而非并行，避免主线程被中间过程污染
   - 配置时需显式约束工具、模型、最大轮次和隔离级别
   - 常见反模式：权限过宽、输出格式不固定、子任务间强依赖

8. 验证闭环构建
   - 层级划分：命令退出码、lint、测试、生产验证
   - 在 Prompt、Skill 和 CLAUDE.md 中显式定义验证标准
   - "说不清楚'怎么算完成'的任务不适合自动完成"

9. CLAUDE.md 编写指南
   - 是协作契约而非知识库，只放每次会话都必须成立的内容
   - 应包含：构建测试命令、架构边界、编码规范、安全限制
   - 不应包含：大段背景、完整API文档、空泛原则

10. 工程化布局建议
    - 完整结构：CLAUDE.md + .claude/rules/ + .claude/skills/ + .claude/agents/
    - 全局约束与项目差异分离，避免互相污染
    - 定期进行配置健康检查

11. 使用阶段演进
    - 工具使用者：关注功能使用
    - 流程优化者：关注协作顺畅，开始编写 CLAUDE.md 和 Skills
    - 系统设计者：关注如何在约束下实现自主运作

12. 常见反模式与修复
    - CLAUDE.md 当 wiki → 污染上下文，关键指令被稀释
    - Skill 大杂烩 → 描述无法稳定触发，工作流冲突
    - 工具太多描述模糊 → 选错工具，schema 挤爆上下文
    - 过度自治 → 多 agent 并行无边界，出错难止损
