## TL;DR
作者分享使用 LLM 编程的工作流程，强调编程从写代码转向系统架构，通过多代理协作构建高质量软件，保持对整体系统的理解。

## Summary
核心主题：作者分享使用大语言模型 (LLM) 编写软件的工作流程，强调 LLM 如何改变编程方式，从编写代码转向系统架构和决策。

关键信息与逻辑脉络：

1.  作者对编程的态度转变
   - 作者原本喜欢编程，但后来发现自己真正喜欢的是创造东西，编程只是实现方式之一
   - LLM 的出现使作者能够专注于创造，而将编码工作交给模型
   - 工程技能并未失效，而是转变为系统架构和决策能力

2.  使用 LLM 编程的优势
   - 代码缺陷率显著低于手动编写
   - 能够保持对整个系统的理解
   - 项目可以扩展到数万行代码而不失可维护性
   - 不同 LLM 模型各有优势，组合使用效果更佳

3.  作者使用 LLM 构建的项目
   - Stavrobot：安全优先的 LLM 个人助手
   - Middle：语音记录吊坠
   - Sleight of hand：艺术性不规则时钟
   - Pine Town：无限多人画布

4.  工具选择：OpenCode
   - 需要支持多个不同公司的模型
   - 需要支持自定义代理之间的自主调用

5.  工作流程：多代理协作
   - 架构师 (Architect)：使用最强模型 (如 Opus)，负责制定详细计划
   - 开发者 (Developer)：使用更高效的模型 (如 Sonnet)，负责实现计划
   - 审查者 (Reviewers)：使用不同模型审查代码，提高质量

6.  工作流程细节
   - 架构师阶段：与 LLM 交互，明确目标和架构，消耗较多时间
   - 开发者阶段：严格按照计划实现，自主权较小
   - 审查阶段：多个独立审查者提供反馈，开发者整合或升级到架构师
   - 整体方法：保持对函数级别以上决策的了解，纠正 LLM 的盲点

7.  失败模式
   - 当对技术不够熟悉时，无法及时发现 LLM 的错误决策
   - 错误决策累积导致代码混乱，难以修复
   - 解决方案：在规划阶段尽可能深入理解技术

8.  实际案例：为 Stavrobot 添加邮件支持
   - 展示了从初始想法到最终实现的完整对话过程
   - 体现了架构师、开发者和审查者的协作
   - 说明了如何通过对话细化计划并确保实现质量
