# 2026 年，我把自己做成了一个 AI
- URL: https://luolei.org/luolei-ai
- Added: 2026-03-06 02:04:46
- Tags: #life #llm

## TL;DR
作者构建个人 AI 分身，整合多平台内容，通过多模型画像与聊天功能，让 AI 理解个人知识结构，实现内容解释权回归，未来将扩展知识库并降低供应商依赖。

## Summary
核心主题：构建个人 AI 分身，实现主动知识结构化，让 AI 完整理解个人内容

关键信息与逻辑脉络：

一、项目背景与动机
1. 作者在互联网留下大量痕迹：博客 300+ 文章、YouTube/B 站视频、X 动态、GitHub 项目、摄影作品等
2. 问题：内容分散在各平台，难以让 AI 完整理解个人形象
3. 核心理念：在生成式 AI 时代，主动构建知识结构比被动被爬虫抓取更重要，本质是拿回对自己内容的解释权

二、"AI 眼中的罗磊"功能
1. 多模型第三方视角画像
   - 使用 6 个 AI 模型（GPT-5.2、Gemini 3、Qwen 3.5 Plus、Kimi K2.5、豆包 Seed 2.0、GLM 5.0）
   - 基于博客文章、X 动态和 GitHub 履历生成
   - 统一 System Prompt 要求客观克制，结论必须有数据支撑

2. 数据来源与处理
   - 博客文章：300+ Markdown 文件，预处理生成结构化摘要
   - X 动态：通过官方 API 抓取，按互动量排序
   - GitHub 履历：项目信息、工作经历、技术栈
   - 数据总量约 11 万字，需预处理压缩才能被大多数模型处理

3. 技术挑战与解决方案
   - 当前大模型上下文窗口限制，需先做摘要压缩
   - 多模型生成提高可信度，不同模型关注点不同

三、AI 数字分身聊天功能
1. 技术栈
   - 框架：Vinext（基于 Vite 的 Next.js 重实现）
   - AI SDK：Vercel AI SDK
   - LLM：通过 OpenAI Compatible API 接入，支持多模型切换
   - 搜索/RAG：自建 @luoleiorg/search-core 包
   - 安全与监控：IP 级速率限制，Telegram Bot 实时通知

2. 工作流程
   - 搜索上下文复用判断：检测追问，复用检索结果
   - 并行搜索：本地关键词搜索 + AI 关键词提取
   - 意图重排：5 类意图分类，按相关度评分
   - 分层 Prompt 构建：核心身份、核心规则、运行时上下文
   - 流式生成与修复：检测响应截断并自动修复
   - 全链路追踪：Token 用量、耗时、引用内容

3. 反幻觉系统工程
   - 来源限制协议：只能使用 Prompt 中的可见信息
   - 数字协议：具体数字必须在文本中出现
   - 履历协议：工作经历以"关于你"为准
   - 链接协议：只允许引用提供的完整 URL

4. 前端设计
   - 移动端全屏、桌面端居中弹窗
   - 键盘 Enter 全局唤起
   - 消息气泡区分用户和 AI
   - 3 秒发送冷却防误触
   - 引用 X 动态自动渲染为卡片

四、思考与展望
1. 数字分身的局限性
   - 基于公开发布内容，缺失未表达的犹豫和想法
   - 呈现的形象可能与真实性格有差异
   - 互联网形象本就是真实自我的投影

2. 未来发展
   - 扩展知识库：增加 YouTube 和 B 站视频内容
   - 数据安全考虑：私密内容的纳入需谨慎
   - 降低依赖性：选择 OpenAI Compatible 接口标准，避免被单一供应商锁定

3. 个人内容创作者的新可能性
   - 从生产内容到构建知识系统
   - 让 AI 成为个人延伸而非替代
   - 随着技术发展，构建个人 AI 分身将变得简单
