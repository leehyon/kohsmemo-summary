# How uv got so fast
- URL: https://nesbitt.io/2025/12/26/how-uv-got-so-fast.html
- Added: 2026-03-12 09:31:25
- Tags: #python #why

## TL;DR
uv 的速度优势主要来自现代标准利用、功能精简和架构决策，Rust 提供了额外但相对较小的性能提升。

## Summary
核心主题：uv 速度优势的关键因素，超越单纯编程语言选择的设计决策

关键信息与逻辑脉络：

1. 速度来源的误解
   - 常见解释："用 Rust 编写"
   - 实际原因：设计决策，而非语言本身
   - 其他 Rust 工具未必同样快速

2. 使 uv 成为可能的标准
   - 问题根源：setup.py 导致的依赖获取困境
   - 解决方案 PEP 系列：
     * PEP 518 (2016)：引入 pyproject.toml 声明构建依赖
     * PEP 517 (2017)：分离构建前端与后端
     * PEP 621 (2020)：标准化 [project] 表
     * PEP 658 (2022)：在 Simple Repository API 中直接提供包元数据
   - 生态系统成熟度：uv 2024 年发布，依赖 2023 年 PyPI 实施的标准

3. uv 放弃的功能（消除带来速度）
   - 不支持 .egg 格式
   - 忽略 pip.conf 配置文件
   - 默认不编译字节码
   - 强制使用虚拟环境
   - 更严格的规范执行
   - 忽略 requires-python 上限
   - 默认仅使用第一个索引源

4. 无需 Rust 的优化
   - HTTP 范围请求获取元数据
   - 并行下载
   - 全局缓存与硬链接
   - 无 Python 的解析
   - PubGrub 解析算法（来自 Dart）

5. Rust 实际贡献
   - 零拷贝反序列化
   - 线程级并行性（绕过 GIL）
   - 无解释器启动开销
   - 紧凑版本表示

6. 设计优于语言
   - uv 速度源于不做什么，而非用什么语言
   - pip 可实现类似优化，但向后兼容性优先
   - 其他包管理器经验：静态元数据、无需代码执行、预解析

结论：uv 的速度优势主要来自现代标准利用、功能精简和架构决策，Rust 提供了额外但相对较小的性能提升
