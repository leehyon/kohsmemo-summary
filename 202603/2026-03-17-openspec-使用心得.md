## TL;DR
OpenSpec 是规格驱动开发工具，将人类意图转化为工程共识，降低 AI 认知负荷，提升执行效率。通过结构化文档管理需求、设计和任务，实现自动化版本控制和变更管理。

## Summary
核心主题：OpenSpec 是一个规格驱动开发工具，旨在将模糊的人类意图转化为结构化的工程共识，降低 AI agent 的认知负荷，提升执行效率和质量

关键信息与逻辑脉络：

1. 核心概念与哲学
   - 规格驱动开发
   - 真相来源 (single source of truth)
   - 意图锁定
   - 行为契约

2. 产物结构
   - proposal.md：需求分析
   - design.md：设计方案
   - tasks.md：任务列表
   - spec.md：系统行为规范

3. 关键功能
   - opsx-propose：创建变更并生成规格
   - opsx-apply：执行实现
   - opsx-verify：验证（可选）
   - opsx-sync：同步（可选）
   - opsx-archive：归档已完成工作
   - opsx-bulk-archive：批量归档

4. 使用方法
   - 初始化项目：`npm install -g @fission-ai/openspec`，`cd projectRoot`，`openspec init`
   - 生成需求、设计、任务和规范：单步路径（/opsx-propose）或分步路径（/opsx-new + /opsx-continue）
   - 审阅和修改生成的 spec
   - 执行：AI agent 根据文档约束进行编码实现
   - 验证（可选）：/opsx-verify
   - 同步（可选）：/opsx-sync
   - 归档：/opsx-archive 或 /opsx-bulk-archive

5. 目录结构
   - changes/：当前进行中的工作
     - archive/：已完成的工作
     - active-feature/：活跃的特性
   - explorations/：处于探索阶段的想法
   - schemas/：spec 模板
   - specs/：主 specs，长期的事实来源

6. 自定义工作流
   - 通过 fork 和修改 schema 模板实现
   - 可针对 bug fix 等轻量级工作流简化流程
   - 可选择跳过 Openspec 直接进行代码修改和 review

7. 配合 AGENTS.md
   - 防止上下文污染和 token 浪费：只读相关 specs 和活跃 changes
   - mono-repo 实践：每个 package 独立管理 openspec 目录
   - 最小化提示词：只包含 AI 无法从代码推断的信息

8. 关于 capability 和 spec
   - spec 是结构化的事实来源，沉淀需求、设计和任务共识
   - capability 是用户可感知的业务能力或系统行为域，长期反复修改的对象
   - 不是细粒度的交互点或一次性缺陷修复
   - Openspec 自动化管理 specs 的版本控制和变更
   - 需要人工审阅和维护 specs

9. 老项目的起步策略
   - 接受无法一键生成主 specs 的现实
   - 三种起步方式：
     1. 作为结构化 change workflow 使用
     2. 为核心、稳定、常改的 capability 补主 specs
     3. 渐进式策略：每次修改到哪个 capability 就补对应的 spec
   - 优先补高价值 capability 的主 specs
   - 不建议优先 spec 化底层实现细节和临时逻辑
