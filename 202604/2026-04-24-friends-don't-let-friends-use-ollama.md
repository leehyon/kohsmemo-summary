# Friends Don't Let Friends Use Ollama
- URL: https://sleepingrobots.com/dreams/stop-using-ollama/
- Added: 2026-04-24 05:14:23
- Tags: #product

## TL;DR
本文批判 Ollama 隐瞒技术来源、性能倒退、误导用户及存在隐私风险，呼吁回归 llama.cpp 或选用更诚信的替代方案。

## Summary
**核心主题**
本文深入批判了热门本地 LLM 运行工具 Ollama，指出其虽凭借易用性占据市场，但存在长期隐匿技术来源、性能倒退、误导用户、转向封闭及云服务等严重问题，呼吁用户回归直接使用 `llama.cpp` 或选择更诚信的替代方案。

**关键信息**
*   **历史与 attribution 问题**：Ollama 本质上是 `llama.cpp` 的封装器，却长期在 README 和二进制分发中隐瞒对 Georgi Gerganov 及 `llama.cpp` 的依赖，违反 MIT 协议的署名要求，视社区贡献为自有资产。
*   **技术架构与性能劣势**：Ollama 曾 Fork `llama.cpp` 后转向基于 `ggml` 的自研后端，导致不仅重新引入旧 Bug，且推理性能显著落后。基准测试显示 `llama.cpp` 比 Ollama 快 1.8 倍，CPU 环境下优势达 30%-50%。
*   **混乱的模型管理**：
    *   **命名误导**：将 DeepSeek 等蒸馏模型 (Distill) 去掉后缀直接作为原版模型发布，导致用户对模型能力产生误解。
    *   **Modelfile 设计缺陷**：引入冗余配置文件而非使用 GGUF 内嵌元数据，导致自动模板检测失败；修改参数需复制整个模型文件（几十 GB），效率极低。
*   **封闭化与隐私风险**：
    *   发布闭源桌面应用且未公开许可证，甚至可能包含 AGPL 依赖。
    *   引入云服务，在未充分告知的情况下将用户数据路由至第三方服务器。
    *   存在安全漏洞 (CVE-2025-51471)，允许恶意服务器窃取认证令牌。
*   **VC 驱动的商业逻辑**：Ollama 遵循典型的“开源包装—获取用户—制造锁定—云变现”路径，通过哈希存储模型增加迁移成本，背离了本地优先 (local-first) 的初衷。
*   **推荐替代品**：推荐直接使用性能最优的 `llama.cpp`，或选择 `Mozilla llamafile`、`LM Studio`、`Jan`、`koboldcpp` 等尊重上游、透明且兼容性更好的工具。

**逻辑脉络**
*   **崛起与隐患**：Ollama 最初通过降低 `llama.cpp` 的使用门槛获得成功，但在获得 VC 融资后，开始刻意抹去对上游项目的认可，埋下道德风险。
*   **技术去依赖失败**：试图摆脱对 `llama.cpp` 的依赖而自研后端，结果导致软件质量下降、性能缩水，技术信誉受损。
*   **用户体验恶化**：在模型注册、命名规范和配置流程上引入不必要的复杂度和误导性操作，将自身置于用户和模型之间作为低效的中间商。
*   **价值观背离**：通过闭源组件、云集成和安全漏洞，逐渐侵蚀了用户对本地隐私和开源精神的信任，暴露了追求商业变现而非服务社区的意图。
*   **最终呼吁**：基于上述道德和技术理由，作者断言 Ollama 不再是值得推荐的工具，真正的价值在于底层引擎 `llama.cpp`，用户应转向更直接、更诚实的替代方案。
