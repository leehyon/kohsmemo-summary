## TL;DR
文章解析 `.claude` 目录结构，通过 `CLAUDE.md`、`settings.json` 及扩展模块定制规则与权限，满足团队与个人需求，打造高效 AI 开发工具。

## Summary
核心主题：深入解析 `.claude` 文件夹的内部结构与配置机制，阐述如何通过定制化指令、规则和权限设置，将 Claude Code 打造成适应团队与个人需求的高效 AI 开发工具。

目录结构与定位
- **双重目录机制**：系统区分项目级与全局级 `.claude` 目录。
  - **项目级 (`.claude/`)**：存放团队配置，包含通用规则、命令和策略，应提交至 Git 版本控制，确保团队一致性。
  - **全局级 (`~/.claude/`)**：存放个人偏好、会话历史及跨项目配置，不纳入版本控制，用于个性化设置。

核心配置文件
- **`CLAUDE.md`**：系统最重要的指令文件，内容直接注入系统提示词。
  - **内容建议**：包含构建测试命令、关键架构决策、注意事项及编码规范。
  - **最佳实践**：精简内容，避免重复 linter 配置，建议控制在 200 行以内以保证指令遵循度。
  - **变体文件**：`CLAUDE.local.md` 用于存放开发者个人偏好，自动被 Git 忽略。
- **`settings.json`**：权限控制中心，定义 Claude 的操作边界。
  - **allow 列表**：定义无需确认即可执行的命令（如 `npm run *`, `git *`）。
  - **deny 列表**：定义完全禁止的操作（如 `rm -rf`, 读取 `.env` 文件）。
  - **未列出的操作**：默认触发确认请求，提供安全网。
  - **变体文件**：`settings.local.json` 用于本地权限覆盖，不提交至代码库。

功能扩展模块
- **`rules/` 目录**：模块化规则管理，解决单文件臃肿问题。
  - 支持通过 YAML frontmatter 设置路径限定，使规则仅在特定文件或目录中生效。
- **`commands/` 目录**：自定义斜杠命令（Slash Commands）。
  - 通过 Markdown 文件创建，文件名即命令名。
  - 支持使用反引号嵌入 Shell 命令（如获取 git diff）及通过 `$ARGUMENTS` 传递参数。
  - 分为项目级命令（`/project:name`）和用户级命令（`/user:name`）。
- **`skills/` 目录**：自动化工作流技能。
  - 与命令不同，技能可根据对话上下文自动触发，无需用户显式调用。
  - 支持捆绑辅助文件，构建复杂的工作流包。
- **`agents/` 目录**：专用子代理人格。
  - 用于特定复杂任务（如代码审查），在隔离的上下文窗口中运行，避免干扰主会话。
  - 可限制工具权限（如仅允许 Read, Grep）并指定使用的模型（如 Haiku 用于轻量级任务）。

实施建议与路径
- **Step 1**：运行 `/init` 初始化并精简 `CLAUDE.md`，确立项目基础规范。
- **Step 2**：配置 `settings.json`，设置基本的权限允许与拒绝列表。
- **Step 3**：创建高频工作流的自定义命令，如代码审查或 Issue 修复。
- **Step 4**：随着项目扩展，将 `CLAUDE.md` 中的复杂规则拆分至 `rules/` 目录，并按需限定作用路径。
- **Step 5**：配置全局 `~/.claude/CLAUDE.md`，固化个人编码风格与偏好。
