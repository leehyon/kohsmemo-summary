# Rust 所有权：C++ RAII 本来想成为的样子
- URL: https://quant67.com/post/rust/ownership-vs-raii/ownership-vs-raii.html
- Added: 2026-04-01 01:42:46
- Tags: #cpp #design

## TL;DR
本文对比 C++ RAII 与 Rust 所有权，指出前者依赖约定易生隐患，后者由编译器强制执行。Rust 通过借用检查彻底杜绝了内存与并发问题，虽 C++ 有工程优势，但 Rust 真正实现了严格的资源管理目标。

## Summary
**核心主题**
本文对比了 C++ 的 RAII（资源获取即初始化）与 Rust 的所有权系统，论证了 Rust 实现了 RAII 理念原本应达到的严格资源管理目标，而 C++ 因历史包袱和向后兼容性，在资源安全上存在诸多漏洞。

**逻辑脉络**
1.  **背景引入**：指出 C++ 的 RAII 虽是伟大发明，但本质上只是一种编程约定，依赖程序员自觉；而 Rust 将其上升为编译器强制执行的法律。
2.  **痛点剖析**：通过五个具体维度对比，详细阐述了 C++ RAII 无法解决的内存安全问题，以及 Rust 如何通过所有权和借用检查在编译期彻底杜绝这些隐患。
3.  **设计哲学**：强调 C++ 是“默认不安全，需要主动防御”，而 Rust 是“默认安全，显式越界”。
4.  **客观评价**：承认 C++ 在生态、灵活性及编译速度等方面的优势，明确指出这些优势与资源安全性属于不同维度。

**关键信息**
*   **本质差异**：C++ RAII 依靠 Convention（约定），编译器不强制；Rust 所有权依靠 Law（法律），违反即编译失败。
*   **Use-After-Move（僵尸对象）**：
    *   C++：对象被移动后处于“valid but unspecified”状态，继续使用会导致未定义行为，但编译器不警告。
    *   Rust：Move 后源变量被“杀掉”，再次使用会直接报编译错误。
*   **悬垂引用（Dangling References）**：
    *   C++：RAII 仅管理资源生命周期，不管引用的生命周期，`string_view` 和迭代器失效常导致悬垂引用。
    *   Rust：借用检查器强制“持有不可变引用时不能修改原数据”，从逻辑上杜绝了悬垂指针和迭代器失效。
*   **共享可变（数据竞争）**：
    *   C++：`shared_ptr` 只保证引用计数的原子性，不保证数据的线程安全，忘记加锁会导致数据竞争。
    *   Rust：通过 `Send` 和 `Sync` trait 在类型系统层面强制并发安全，跨线程共享可变数据必须使用 `Mutex` 等同步机制。
*   **异常安全**：
    *   C++：构造函数抛异常时析构函数不调用，极易造成半构造对象的资源泄漏。
    *   Rust：没有可恢复异常，使用 `Result` 和 `?` 处理错误，保证任何步骤失败时，已构造的变量都会被正确 Drop。
*   **逃生舱门**：
    *   C++：提供多种默认开启的不安全操作（如裸指针、`.get()`），安全依赖团队规范。
    *   Rust：`unsafe` 块默认关闭且显式标记，便于代码审计，风险集中在少数经过审查的底层代码。
*   **C++ 的优势领域**：生态成熟度（40 年积累）、自定义分配器（精细内存控制）、编译速度、C 语言互操作成本、模板元编程灵活性。这些属于工程和灵活性优势，与安全性无关。
