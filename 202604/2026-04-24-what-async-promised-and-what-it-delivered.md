# What Async Promised and What it Delivered
- URL: https://causality.blog/essays/what-async-promised/
- Added: 2026-04-24 02:09:44
- Tags: #language

## TL;DR
异步编程从 Callbacks 演变至 Async/Await，虽解决了并发与局部语法问题，却引发了函数着色、生态分裂与顺序陷阱等全局成本，促使部分语言探索轻量级线程等替代方案。

## Summary
**核心主题**
本文回顾了异步编程从 Callbacks 到 Promises 再到 Async/Await 的演变历程，指出每一波浪潮在解决高并发（C10K）资源限制和前代语法缺陷的同时，都引入了新的结构性成本与复杂性。最终，Async/Await 虽然优化了局部代码的可读性，却带来了全局性的“函数着色”问题与“顺序陷阱”，加剧了生态系统碎片化与维护负担。

**关键信息**
1.  **OS 线程的局限与 Callbacks 的兴起：** 为解决 C10K 问题，Callbacks 利用非阻塞 I/O 和事件循环解决了线程资源耗尽问题，但造成了控制流反转和“回调地狱”，且难以处理错误传播和取消操作。
2.  **Promises 的改进与局限：** Promises 将异步操作对象化，改善了组合性和错误处理，但引入了静默错误、一次性限制，以及同步与异步返回类型的初步分裂。
3.  **Async/Await 的普及与代价：** Async/Await 极大改善了线性异步代码的编写体验，使其看起来像同步代码，被主流语言广泛采用。然而，它固化了“函数着色”问题，即同步和异步函数相互隔离，导致代码修改传染性扩散、生态系统分裂（如 Rust 的 Tokio 与 async-std 不兼容）以及维护成本翻倍。
4.  **顺序陷阱与并发性丧失：** Async/Await 的语法容易误导开发者编写顺序代码，从而错失并行执行的机会。挖掘并行性需要手动分析依赖并打破线性语法，增加了认知负担。
5.  **新类别 Bug：** 异步环境引入了线程中不存在的新问题，如异步 Rust 中的“Futurelocks”，即持有锁的 Future 停止被轮询导致的死锁。
6.  **替代方案的探索：** Go 的 Goroutines 和 Java 的 Project Loom（虚拟线程）选择通过轻量级线程而非语法层面的 Async/Await 来解决并发，从而避免了函数着色带来的生态分裂。

**逻辑脉络**
1.  **起因：** 操作系统线程高昂的内存和调度成本无法应对 C10K 级别的并发连接。
2.  **第一波：** Callbacks 通过非阻塞 I/O 解决了资源问题，但牺牲了代码结构和控制流的可读性。
3.  **第二波：** Promises 通过对象化和链式调用解决了嵌套和部分错误处理问题，但引入了组合复杂性和类型混淆。
4.  **第三波：** Async/Await 通过语法糖解决了编写体验，使异步代码看似同步，但未能解决底层同步与异步的隔离。
5.  **矛盾爆发：** 函数着色导致代码库中同步与异步逻辑的撕裂，修改具有传染性；同时，顺序化语法掩盖了并发潜力，导致性能优化困难。
6.  **总结：** 每一轮技术演进都在局部改善了开发体验，却在全局累积了“税负”。这揭示了仅关注“如何管理并发执行”的抽象方法会不断衍生新问题，促使部分语言设计者转向轻量级线程等不同路径。
