# Harness engineering for coding agent users

## TL;DR
本文探讨面向编码代理的 Harness 工程，利用 Guides 和 Sensors 实现前馈引导与反馈自纠。该系统将人类经验显性化，提升 AI 代码生成质量，减轻审查负担并建立信任。

## Summary
### 核心主题

本文探讨了面向编码代理用户的 **Harness 工程**，即如何构建一套外部的控制系统，通过“指引”与“传感器”的组合来约束和引导 AI 编码代理。其核心目标是将人类开发者的隐性经验显性化，通过前馈引导和反馈自纠机制，提高 AI 代码生成的首次正确率，减少人工审查负担，并建立对 AI 输出结果的信任。

### 关键信息

**1. Harness 的定义与构成**
*   **公式**：Agent = Model + Harness。
*   **边界**：对于用户而言，Harness 是指除模型外，用户为特定用例构建的外部控制系统。
*   **组件**：包含 **Guides (指引)** 和 **Sensors (传感器)**。

**2. 两大核心机制**
*   **Guides (前馈控制)**：在代理行动前预测并引导行为，提高首次生成质量。例如：规范文档、API 文档、How-to 指南。
*   **Sensors (反馈控制)**：在代理行动后观察结果，提供信号触发自纠。例如：Linter、静态分析、AI 代码审查。
*   **执行类型**：
    *   **计算型**：确定性强、速度快 (CPU)，如 Linter、类型检查、单元测试。
    *   **推理型**：语义分析、非确定性 (GPU/NPU)，如 AI 代码审查、LLM 作为评审员。

**3. 时机与布局策略**
*   **质量左移**：将检查尽可能置于开发流程的左侧。快速检查（如 Linter、基础测试）应在提交或集成前运行；昂贵的检查（如变异测试、全面审查）应在集成后的流水线中运行。
*   **持续监控**：除变更生命周期外，还需设置传感器监控代码库的持续漂移（如死代码检测）和运行健康状况（如 SLO 监控、日志异常分析）。

**4. 监管维度分类**
*   **可维护性 Harness**：监管内部代码质量，如结构、风格、重复代码、测试覆盖率。这是目前最容易实现的部分，主要依赖计算型工具。
*   **架构适应性 Harness**：监管架构特征，如性能要求、日志标准、模块边界。通过架构适应度函数 来实现。
*   **行为 Harness**：监管功能正确性。目前主要依赖功能规范（前馈）和 AI 生成的测试套件（反馈），但仍需大量人工监督，是当前的难点所在。

**5. 实践挑战与演进**
*   **可 Harness 性**：强类型语言和清晰架构边界的代码库更容易构建 Harness。遗留代码因技术债务，构建 Harness 难度更大。
*   **Harness 模板**：未来可能会出现针对常见服务拓扑（如 CRUD 服务、事件处理器）的标准化 Harness 模板，包含特定的指引和传感器集。
*   **人类角色**：人类负责“引导” Harness 的迭代。当问题重复出现时，改进指引和传感器。Harness 的目标是显性化人类经验，而非完全取代人类输入。

### 逻辑脉络

**1. 概念界定与范围聚焦**
文章首先定义了 Harness 在编码代理特定语境下的含义，将其区分为构建者内置部分和用户自定义部分，并指出用户构建外部 Harness 对于提升代理表现至关重要。

**2. 机制解析与类型划分**
接着，文章深入剖析了 Harness 的运作机制，将其拆解为前馈指引和反馈传感器，并进一步区分了计算型（确定、快速）和推理型（语义、昂贵）两种执行方式，强调两者结合的必要性。

**3. 实施策略与生命周期管理**
随后，文章探讨了在开发流程中如何部署这些机制，提出“质量左移”原则，详细规划了从代码编写到集成部署，再到持续运维的全生命周期传感器布局。

**4. 维度分类与能力评估**
文章进而将 Harness 的监管对象细分为可维护性、架构适应性和行为三个维度，评估了当前工具在不同维度上的成熟度，指出行为验证仍是短板。

**5. 宏观视角与未来展望**
最后，文章从代码库特性、模板化趋势以及人类在引导循环中的角色进行了宏观讨论，提出了 Harness 作为一种新兴工程实践的开放问题和未来方向。
