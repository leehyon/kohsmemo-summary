# Do You Even Need a Database
- URL: https://www.dbpro.app/blog/do-you-even-need-a-database
- Added: 2026-04-17 01:52:17
- Tags: #engineering #why

## TL;DR
文章探讨了早期使用文件系统替代数据库的可行性。测试显示，内存映射等简单方案性能优异，SQLite 也足以支撑大流量。结论是应按需选择，无需过早引入数据库，简单文件方案往往已足够。

## Summary
**核心主题**
探讨在软件开发早期或特定规模下，直接使用文件系统（如 JSONL）替代传统数据库的可行性与性能优势，主张应根据实际规模和需求选择存储方案，而非默认引入数据库。

**关键信息**
*   **测试背景与方法**：使用 Go、Bun (TypeScript) 和 Rust 构建简单的 HTTP 服务器，对比四种存储策略（线性扫描、内存映射、磁盘二分查找、SQLite）在不同数据量级（1万至 100万记录）下的 `GET` 请求性能。
*   **存储策略性能对比**：
    *   **线性扫描**：每次请求全量扫描文件，复杂度为 O(n)，性能随数据量增长线性下降，仅适用于极小规模。
    *   **内存映射**：启动时将数据加载至 Hash Map，读取复杂度为 O(1)，吞吐量极高（约 9.7万至 16.9万 req/s），但受限于 RAM 容量。
    *   **磁盘二分查找**：通过排序数据文件和固定宽度索引实现 O(log n) 查找，无需全量加载内存，吞吐量（约 4万 req/s）优于 SQLite，且性能随数据增长保持稳定。
    *   **SQLite**：性能稳定且良好（约 2.5万 req/s），略逊于手写二分查找，但提供了完整的 SQL 功能支持。
*   **规模与容量分析**：将性能指标转化为日活用户 (DAU) 承载能力，SQLite 单机即可支撑约 9000 万 DAU。大多数应用（如万级用户的 SaaS）的实际负载远低于此阈值，无需过早引入数据库。
*   **结论**：对于许多应用，简单的文件方案已足够。若未来需要迁移，JSONL 格式也极易导入数据库。

**逻辑脉络**
*   **提出观点**：数据库本质上是文件，对于许多应用（特别是早期阶段），直接操作自定义文件可能比使用数据库更合适。
*   **技术验证**：通过多语言、多策略的基准测试，证明在简单的 ID 查询场景下，内存映射和磁盘二分查找等自定义文件方案，在性能上往往优于或相当于标准数据库。
*   **场景界定**：明确只有在以下情况发生时，才真正有必要引入数据库：数据量超过单机 RAM 容量、需要按非 ID 字段进行复杂查询、需要联表操作、需要多进程并发写入一致性、以及需要跨实体的 ACID 事务支持。
